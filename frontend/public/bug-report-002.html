BUG-002

Hardcoded admin credentials exposed on production login page allow unauthorized access to admin panel

OPEN
▶ See Live

Reproducibility 100%
Reporter

Environment Any browser · Any device · pyavchik.space/admin/login · SlotsOne Admin v1.0

Description

The admin panel login page at /admin/login displays a plaintext hint containing valid admin credentials: admin@slotsone.com / admin123. Any visitor can read these credentials directly from the page and gain full administrative access to the admin panel — including player management, wallet top-ups, transaction history, and audit logs.

OWASP Classification

A07:2021 — Identification and Authentication Failures. Hardcoded credentials displayed in a production UI constitute a critical authentication bypass. This also relates to A05:2021 — Security Misconfiguration (debug/demo features left enabled in production).

Preconditions

  • Access to pyavchik.space (public internet)
  • A web browser

Steps to Reproduce

1

Navigate to pyavchik.space/admin/login

2

Read the text below the Sign in button

It reads: "Demo: admin@slotsone.com / admin123"

3

Enter admin@slotsone.com in the Email field and admin123 in the Password field

4

Click Sign in

You are now logged into the admin panel with full access.

Expected vs Actual Result

✓ Expected

The login page shows only email and password fields. No credentials, hints, or demo text are displayed. Authentication requires knowledge of privately provisioned credentials.

✗ Actual

A <p> element below the submit button displays working admin credentials in plaintext: "Demo: admin@slotsone.com / admin123". Anyone visiting the page can log in as admin.

Source Code & Rendered Output

S Source — admin/app/login/page.tsx:95-97
// Lines 95-97 of login page component <p className="text-center text-xs text-muted-foreground"> Demo: admin@slotsone.com / admin123 </p>
This JSX is unconditionally rendered on the production login page with no environment check.
H Rendered HTML — /admin/login
<!-- Visible to any visitor --> <p class="text-center text-xs text-muted-foreground"> Demo: admin@slotsone.com / admin123 </p>
The credentials are rendered as regular DOM text — visible in page source, DevTools, screen readers, and crawlers.

Evidence

CREDENTIALS EXPOSED
The admin login page at /admin/login displays working credentials below the Sign in button. Any visitor can read and use them.

Root Cause Analysis

A demo credential hint was hardcoded into the login page component during development to simplify local testing. The hint was never gated behind an environment variable or removed before deployment to production. The credentials are real — they match accounts seeded via npm run db:seed and grant full admin panel access.

1. Developer adds demo credential hint during local development for convenience
2. Hint is hardcoded in JSX with no process.env guard or build-time stripping
3. Code is deployed to production — hint renders on the public login page VULN
4. Seeded admin accounts (admin@slotsone.com, superadmin@slotsone.com) use the displayed password VULN
5. Any visitor can read credentials and log in with full admin privileges VULN
PROPOSED FIX

Remove hardcoded credentials from the login page

Delete the <p> element that displays the demo hint. Additionally, change the seeded admin passwords and rotate any sessions that may have been created with the exposed credentials.

Additional remediation steps

1. Remove the <p> element from admin/app/login/page.tsx:95-97
2. Change passwords for admin@slotsone.com and superadmin@slotsone.com in production DB
3. Invalidate all existing admin sessions / refresh tokens
4. Review audit logs for unauthorized admin actions during the exposure window
