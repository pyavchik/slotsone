Rest Assured — API Testing REST ASSURED

24 API tests across three test classes validate authentication, game spin mechanics, and history endpoints. A shared BaseApiTest configures the Rest Assured request specification with JSON content type, Allure logging filters, and auth helper methods. Responses are deserialized into Jackson DTOs and validated with both Hamcrest matchers and AssertJ fluent assertions.

BaseApiTest — shared request spec with Allure integration:

public abstract class BaseApiTest {
    protected RequestSpecification baseSpec;

    @BeforeClass
    public void setupApi() {
        RestAssured.baseURI = config.baseUrl();
        RestAssured.basePath = config.apiBasePath();

        baseSpec = new RequestSpecBuilder()
                .setContentType(ContentType.JSON)
                .addFilter(new AllureRestAssured())
                .addFilter(new RequestLoggingFilter())
                .addFilter(new ResponseLoggingFilter())
                .build();
    }

    protected RequestSpecification authSpec() {
        return given().spec(baseSpec)
                .header("Authorization", "Bearer " + accessToken);
    }
}

Example — SpinApiTest with idempotency validation:

@Epic("API")  @Feature("Game Spin")
public class SpinApiTest extends BaseApiTest {

    @Test(description = "Spin returns valid response with outcome and balance")
    @Severity(SeverityLevel.BLOCKER)
    public void testSpinReturnsValidResponse() {
        Response spin = authSpec()
                .body(defaultSpinBody())
                .post("/spin")
                .then()
                .statusCode(200)
                .body("spin_id", notNullValue())
                .body("outcome.reel_matrix", notNullValue())
                .extract().response();

        assertThat(spin.jsonPath().getList("outcome.reel_matrix"))
                .hasSize(5);
    }

    @Test(description = "Duplicate spin requests are idempotent")
    @Severity(SeverityLevel.CRITICAL)
    public void testIdempotentSpin() {
        String key = UUID.randomUUID().toString();

        String first = authSpec()
                .header("Idempotency-Key", key)
                .body(body).post("/spin")
                .then().statusCode(200)
                .extract().jsonPath().getString("spin_id");

        String second = authSpec()
                .header("Idempotency-Key", key)
                .body(body).post("/spin")
                .then().statusCode(200)
                .extract().jsonPath().getString("spin_id");

        assertThat(first).isEqualTo(second);
    }
}

Additional API test scenarios include JSON Schema validation via matchesJsonSchemaInClasspath(), negative cases (400 for invalid input, 401 for missing/invalid tokens, 409 for duplicates), and balance arithmetic assertions with isCloseTo() offsets.

Selenide — UI Test Automation SELENIDE

Selenide wraps Selenium WebDriver with concise, auto-waiting APIs. 13 UI tests cover login, game canvas interaction, and history table navigation. The Page Object Model pattern encapsulates all locators and actions behind descriptive @Step-annotated methods for clean Allure reports and easy maintenance.

LoginPage — Page Object with fluent Selenide API and Allure steps:

public class LoginPage {

    private final SelenideElement passwordInput = $("#auth-password");
    private final SelenideElement submitButton  = $("button.auth-submit");
    private final SelenideElement ageCheckbox   = $("#auth-age");
    private final SelenideElement errorMessage  = $(".auth-error");

    @Step("Open login page")
    public LoginPage open() {
        Selenide.open("/login");
        authCard.shouldBe(visible);
        return this;
    }

    @Step("Register as {email}")
    public LobbyPage registerAs(String email, String password) {
        switchToRegister();
        enterEmail(email);
        enterPassword(password);
        confirmAge();
        submit();
        return new LobbyPage();
    }

    @Step("Verify error message is displayed")
    public LoginPage verifyErrorDisplayed() {
        errorMessage.shouldBe(visible);
        return this;
    }
}

Selenoid — Containerized Browser Grid SELENOID

Selenoid replaces traditional Selenium Grid with lightweight Docker containers. Each test session spins up a fresh browser container, ensuring isolation and reproducibility. VNC enables live session viewing; video recording captures every test run for debugging. BaseUiTest auto-detects whether Selenoid is enabled and configures DesiredCapabilities accordingly.

BaseUiTest — dynamic Selenoid/local browser configuration:

public abstract class BaseUiTest {

    @BeforeSuite
    public void setupAllureListener() {
        SelenideLogger.addListener("AllureSelenide",
                new AllureSelenide().screenshots(true).savePageSource(true));
    }

    @BeforeClass
    public void setupBrowser() {
        Configuration.baseUrl = config.baseUrl();
        Configuration.browser = config.browser();
        Configuration.headless = config.browserHeadless();

        if (config.selenoidEnabled()) {
            Configuration.remote = config.selenoidUrl();

            DesiredCapabilities caps = new DesiredCapabilities();
            caps.setCapability("selenoid:options", new HashMap<>() {{
                put("enableVNC", config.selenoidVncEnabled());
                put("enableVideo", config.selenoidVideoEnabled());
                put("enableLog", true);
            }});
            Configuration.browserCapabilities = caps;
        }
    }
}

TestNG — Suite Configuration TESTNG

Five TestNG XML suites define how tests are grouped and parallelized. API tests run in parallel across 3 threads (no browser needed), UI tests run sequentially (shared browser state), and BDD tests have their own suite. Maven profiles map 1:1 to suites, allowing mvn test -Papi or mvn test -Pall from the command line.

testng-all.xml — full suite with parallelization strategy:

<suite name="SlotsOne Full Test Suite"
       parallel="tests" thread-count="2">
  <listeners>
    <listener class-name="io.qameta.allure.testng.AllureTestNg"/>
  </listeners>

  <!-- API tests: fast, parallel by class -->
  <test name="API Tests" parallel="classes" thread-count="3">
    <classes>
      <class name="com.slotsone.qa.api.AuthApiTest"/>
      <class name="com.slotsone.qa.api.SpinApiTest"/>
      <class name="com.slotsone.qa.api.HistoryApiTest"/>
    </classes>
  </test>

  <!-- UI tests: sequential, require browser -->
  <test name="UI Tests" parallel="false" preserve-order="true">
    <classes>
      <class name="com.slotsone.qa.ui.LoginUiTest"/>
      <class name="com.slotsone.qa.ui.GameUiTest"/>
      <class name="com.slotsone.qa.ui.HistoryUiTest"/>
    </classes>
  </test>
</suite>

CI/CD — Jenkins + GitLab CI CI/CD

Two CI/CD pipelines are configured for maximum flexibility. Jenkins uses a parameterized declarative pipeline with optional Selenoid startup. GitLab CI leverages Selenoid as a Docker service for parallel test stages. Both pipelines aggregate results into a unified Allure report.

Jenkinsfile — parameterized multi-stage pipeline:

pipeline {
    parameters {
        string(name: 'BASE_URL', defaultValue: 'http://localhost:3000')
        choice(name: 'TEST_SUITE', choices: ['all', 'api', 'ui', 'bdd'])
        booleanParam(name: 'USE_SELENOID', defaultValue: true)
    }

    stages {
        stage('Start Selenoid') {
            when { expression { params.TEST_SUITE in ['all', 'ui'] && params.USE_SELENOID } }
            steps { sh 'docker compose -f docker-compose.selenoid.yml up -d' }
        }
        stage('API Tests')  { sh 'mvn test -Papi -Dbase.url=${BASE_URL}' }
        stage('UI Tests')   { sh 'mvn test -Pui -Dselenoid.enabled=true' }
        stage('BDD Tests')  { sh 'mvn test -Pbdd -Dselenoid.enabled=true' }
    }
    post {
        always {
            allure results: [[path: "${ALLURE_RESULTS}"]]
            sh 'docker compose -f docker-compose.selenoid.yml down || true'
        }
    }
}

.gitlab-ci.yml — parallel stages with Selenoid service:

Configuration — Owner + Maven Properties CONFIG

The Owner library provides type-safe configuration with a layered resolution order: system properties → environment variables → test.properties file. This means Maven -D flags override defaults seamlessly, enabling CI environments to inject -Dbase.url, -Dselenoid.url, etc. without touching config files. 
@Sources({"system:properties", "system:env", "classpath:test.properties"})
public interface TestConfig extends Config {

    @Key("base.url")          @DefaultValue("http://localhost:3000")
    String baseUrl();

    @Key("selenoid.enabled")   @DefaultValue("false")
    boolean selenoidEnabled();

    @Key("selenoid.url")       @DefaultValue("http://localhost:4444/wd/hub")
    String selenoidUrl();

    @Key("browser.headless")   @DefaultValue("true")
    boolean browserHeadless();

    static TestConfig getInstance() {
        return ConfigFactory.create(TestConfig.class,
                System.getProperties(), System.getenv());
    }
}

Environment override examples:

# Target production with Selenoid
mvn test -Pall \
  -Dbase.url=https://pyavchik.space \
  -Dselenoid.enabled=true \
  -Dselenoid.url=http://selenoid:4444/wd/hub

# Local headless, API only
mvn test -Papi -Dbase.url=http://localhost:3000

Quick Start

Prerequisites: Java 17+, Maven 3.9+, Docker (for Selenoid)

# Run API tests
mvn clean test -Papi

# Run UI tests via Selenoid
docker compose -f docker-compose.selenoid.yml up -d
mvn clean test -Pui -Dselenoid.enabled=true
# Watch live: http://localhost:8080 (Selenoid UI)

# Run BDD scenarios
mvn clean test -Pbdd

# Run everything
mvn clean test -Pall

# Generate Allure report
mvn allure:serve
