Pytest fixtures & typed Requests client PYTEST

A small ApiClient wraps requests.Session with timing, bearer auth, and an expect_ok() helper that fails fast with a clear message. Session-scoped settings come from environment variables (or a .env file) so the same suite runs unchanged against localhost, staging, or production. The authed_api fixture registers a fresh user per test — no shared state, no flakiness from leftover data.

framework/api_client.py — thin Requests wrapper:

class ApiClient:
    def __init__(self, base_url: str | None = None, token: str | None = None) -> None:
        self.base_url = (base_url or SETTINGS.api_base_url).rstrip("/")
        self._session = requests.Session()
        self._token = token

    def request(self, method, path, *, json_body=None, params=None, headers=None, token=None):
        merged = {"Accept": "application/json"}
        if json_body is not None:
            merged["Content-Type"] = "application/json"
        eff_token = token if token is not None else self._token
        if eff_token:
            merged["Authorization"] = f"Bearer {eff_token}"
        if headers: merged.update(headers)

        start = time.perf_counter()
        resp = self._session.request(method.upper(), f"{self.base_url}{path}",
            json=json_body, params=params, headers=merged,
            timeout=SETTINGS.request_timeout_s)
        elapsed_ms = (time.perf_counter() - start) * 1000.0

        body = resp.json() if resp.headers.get("content-type", "").startswith("application/json") else resp.text
        return ApiResponse(status=resp.status_code, body=body, headers=resp.headers, elapsed_ms=elapsed_ms)

Contract validation against the live OpenAPI doc JSON SCHEMA

Every successful response is validated against backend/openapi.json with Draft202012Validator. Drift between the implementation and the published contract fails CI before partners or the frontend ever see it. One subtle gotcha: OpenAPI 3.0’s nullable: true keyword does not exist in JSON Schema 2020-12, so the loader normalises it to a type union (type: [..., "null"]) on load. This caught a real bug during initial bring-up — the spec advertised outcome.bonus_triggered as a non-nullable object, but production returned null on every base-game spin.

framework/schemas.py — OpenAPI 3.0 nullable normaliser:

def _normalize_nullable(node):
    """Translate OpenAPI 3.0 `nullable: true` into JSON Schema's `type: [..., "null"]`."""
    if isinstance(node, dict):
        if node.pop("nullable", False):
            t = node.get("type")
            if isinstance(t, str):
                node["type"] = [t, "null"]
            elif isinstance(t, list) and "null" not in t:
                node["type"] = [*t, "null"]
            # type-less + nullable → empty schema accepts anything (incl. null)
        for v in node.values(): _normalize_nullable(v)
    elif isinstance(node, list):
        for v in node: _normalize_nullable(v)
    return node

def assert_matches(component_name, payload):
    errors = sorted(validator_for(component_name).iter_errors(payload),
                     key=lambda e: list(e.path))
    if not errors: return
    formatted = "\n".join(f"  - {'/'.join(map(str, e.path)) or '<root>'}: {e.message}"
                            for e in errors)
    raise AssertionError(f"Payload does not match schema {component_name}:\n{formatted}")

Provably-fair HMAC re-derivation CRYPTO

The operator commits a SHA-256 hash of server_seed before any spin. Each round’s RNG seed is then derived as HMAC-SHA256(server_seed, "<client_seed>:<nonce>")[:4]. After seed rotation, the previously-active server_seed is revealed. The test re-derives the round seed locally with that revealed value and asserts that sha256(revealed) equals the commitment stored on the round — a mathematical guarantee that the operator could not have post-hoc tampered with the outcome.

framework/provably_fair.py — mirrors the backend's derivation:

def sha256_hex(data: str) -> str:
    return hashlib.sha256(data.encode("utf-8")).hexdigest()

def derive_round_seed(server_seed: str, client_seed: str, nonce: int) -> int:
    """HMAC-SHA256(server_seed, "<client_seed>:<nonce>"), first 4 bytes → uint32 RNG seed."""
    msg = f"{client_seed}:{nonce}".encode("utf-8")
    digest = hmac.new(server_seed.encode("utf-8"), msg, hashlib.sha256).digest()
    return int.from_bytes(digest[:4], byteorder="big", signed=False)

def server_seed_matches_hash(revealed: str, published_hash: str) -> bool:
    return sha256_hex(revealed).lower() == published_hash.lower()

The test that closes the loop — spin, rotate, re-derive, verify:

@pytest.mark.regression
@pytest.mark.provably_fair
def test_round_seed_can_be_independently_re_derived(authed_api):
    init = authed_api.post("/api/v1/game/init", json_body={...}).expect_ok().body
    spin = authed_api.post("/api/v1/spin", json_body={...}).expect_ok().body

    rotated  = authed_api.post("/api/v1/provably-fair/rotate").expect_ok().body
    revealed = (rotated.get("previous") or {}).get("server_seed")
    detail   = authed_api.get(f"/api/v1/history/{spin['spin_id']}").expect_ok().body
    pf       = detail["provably_fair"]

    # 1) Revealed seed must hash to the commitment stored on the round.
    assert sha256_hex(revealed) == pf["server_seed_hash"]

    # 2) Independently re-derive the per-round RNG seed.
    seed = derive_round_seed(revealed, pf["client_seed"], int(pf["nonce"]))
    assert 0 <= seed < 2**32

Playwright UI tests with Page Objects PLAYWRIGHT

UI tests deliberately stay thin — the framework seeds state via the API (faster, deterministic) and uses Playwright only to drive flows that genuinely require a browser. Page Objects expose a small, intention-revealing API (visit(mode="register"), fill_credentials(), submit()) so test bodies read like specification narratives.

tests/ui/pages/auth_page.py — fluent Page Object:

class AuthPage:
    LOGIN_PATH = "/login"
    REGISTER_PATH = "/register"

    def visit(self, *, mode: str = "login") -> "AuthPage":
        path = self.REGISTER_PATH if mode == "register" else self.LOGIN_PATH
        self.page.goto(f"{self.base_url}{path}")
        expect(self.page.locator("#auth-email")).to_be_visible()
        return self

    def fill_credentials(self, email, password):
        self.page.locator("#auth-email").fill(email)
        self.page.locator("#auth-password").fill(password)
        return self

    def submit(self): self.page.get_by_role("button", name="Create Account").click(); return self
    def submit_login(self): self.page.get_by_role("button", name="Login").click(); return self

CI/CD — GitHub Actions with ephemeral keys CI/CD

The workflow boots a Postgres service container, builds the backend, mints a fresh RS256 keypair per run (so committed keys never touch the repo), starts the backend, and runs Pytest with parallel workers and one auto-retry. Allure results upload as a build artifact for later inspection. Triggers on PR for any path under qa-python/, backend/, or frontend/.

.github/workflows/qa-python.yml:

name: qa-python
on:
  pull_request:
    paths: ["qa-python/**", "backend/**", "frontend/**", ".github/workflows/qa-python.yml"]

jobs:
  api:
    runs-on: ubuntu-latest
    services:
      postgres: { image: postgres:16-alpine, ports: ["5432:5432"], ... }
    steps:
      - name: Generate ephemeral RS256 JWT keypair
        run: |
          openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:2048 -out jwt_private.pem
          openssl rsa -in jwt_private.pem -pubout -out jwt_public.pem
          { echo 'JWT_PRIVATE_KEY<<EOF'; cat jwt_private.pem; echo EOF; } >> "$GITHUB_ENV"
          { echo 'JWT_PUBLIC_KEY<<EOF';  cat jwt_public.pem;  echo EOF; } >> "$GITHUB_ENV"

      - name: Start backend
        run: |
          nohup node dist/index.js > backend.log 2>&1 &
          for i in $(seq 1 40); do
            curl -sf http://localhost:3001/ready >/dev/null && { echo backend up; exit 0; }
            sleep 1
          done; exit 1

      - name: Run API tests
        run: pytest tests/api -n auto --reruns 1 --alluredir=allure-results

      - uses: actions/upload-artifact@v4
        with: { name: qa-python-allure-results, path: qa-python/allure-results }

AWS integrations — boto3, moto, LocalStack AWS

The framework speaks boto3 against three interchangeable backends: real AWS, a LocalStack container, or an in-process moto ThreadedMotoServer. The choice is made by environment: if AWS_ENDPOINT_URL is set, boto3 talks to that endpoint; if CI_AWS_REAL=1, it uses ambient AWS credentials; otherwise a moto server is spun up on a free port for the test session. Same test code runs everywhere — what changes is the trust level of the signal. AWS-tagged tests are gated behind @pytest.mark.aws so the rest of the suite runs without any AWS infrastructure.

framework/aws/s3_assertions.py — bounded waiter for async uploads:

def wait_for_object(bucket, key, *, timeout_s=10.0, poll_interval_s=0.5):
    """Poll until s3://bucket/key exists or the timeout elapses.

    Use when an upload is asynchronous (Lambda, queue worker) and the test
    cannot synchronise on the producer."""
    deadline = time.monotonic() + timeout_s
    while time.monotonic() < deadline:
        head = _head_object(bucket, key)
        if head is not None:
            return head
        time.sleep(poll_interval_s)
    raise AssertionError(f"S3 object did not appear within {timeout_s}s: s3://{bucket}/{key}")

tests/aws/test_sqs_event_published.py — predicate-based queue assertion:

@pytest.mark.aws
@pytest.mark.smoke
def test_published_event_lands_with_expected_shape(sqs_queue):
    drain_queue(sqs_queue)
    event_id = str(uuid.uuid4())
    payload = {"event_type": "wallet_credited", "event_id": event_id,
               "user_id": "user-abc-123", "amount_cents": 500, "currency": "USD"}
    aws_client("sqs").send_message(QueueUrl=sqs_queue, MessageBody=json.dumps(payload))

    matched = assert_message_in_queue(
        sqs_queue,
        predicate=lambda body: isinstance(body, dict)
            and body.get("event_id") == event_id
            and body.get("event_type") == "wallet_credited",
        timeout_s=5,
    )
    body = json.loads(matched["Body"])
    assert body["amount_cents"] == 500

tests/aws/conftest.py — auto-selects backend so any developer can run AWS tests:

@pytest.fixture(scope="session", autouse=True)
def _aws_session_setup():
    # 1) CI_AWS_REAL=1     → real AWS, ambient creds
    # 2) AWS_ENDPOINT_URL → LocalStack / staging / etc.
    # 3) neither          → in-process moto server (zero infra)
    if os.getenv("CI_AWS_REAL") == "1": yield; return
    if os.getenv("AWS_ENDPOINT_URL"): yield; return

    from moto.server import ThreadedMotoServer
    server = ThreadedMotoServer(ip_address="127.0.0.1", port=0)
    server.start()
    host, port = server.get_host_and_port()
    os.environ["AWS_ENDPOINT_URL"] = f"http://{host}:{port}"
    reset_clients()
    try: yield
    finally: server.stop()

Honest framing: these tests demonstrate AWS-aware testing patterns — bucket/queue/log post-conditions, async-upload waiters, predicate matchers, endpoint portability. They are not a substitute for years of production AWS exposure (IAM, VPC, multi-region, cost models). What they show is that I know which observability surfaces a QA layer cares about and how to assert against them in CI.

Why this design

A short tour through the deliberate choices.

No business-logic mocks. Every test hits a real backend with a real Postgres — CI spins both up. Mocked tests passing while a migration silently breaks production is a failure mode this codebase has explicitly chosen to avoid.

Contract is the source of truth. The framework reads backend/openapi.json directly. There is no parallel set of Python DTOs to drift; the only place a response shape lives is the published spec.

Cryptographic verification, not just status-code checks. The provably-fair tests prove a property the operator cannot fake by adjusting their server — they would have to break SHA-256 or HMAC.

Healthcare-pivot ready. For an FHIR / HL7v2 / DICOM context, the same architecture applies: swap schemas.py over to FHIR StructureDefinition JSON, add an OAuth2 token-exchange step in the ApiClient, and layer pytest-bdd for scenario-driven compliance documentation. The framework split (config / client / contract / domain helpers) was chosen so this swap stays surgical.
