Test Scenarios

Scenario 1: Baseline Performance PASS
Configuration 10 concurrent users, 5-minute duration, 30s ramp-up
Target Avg response <100ms, P95 <200ms, 0% errors
Results Avg 32ms, P95 78ms, 0% errors
Throughput 185 req/sec
Scenario 2: Peak Load Simulation PASS
Configuration 50 concurrent users, 10-minute duration, 60s ramp-up
Target Avg response <200ms, P95 <500ms, <1% errors
Results Avg 47ms, P95 142ms, 0.02% errors
Throughput 820 req/sec
Scenario 3: Stress Test (Breaking Point) DEGRADED
Configuration 200 concurrent users, 5-minute duration, 30s ramp-up
Target Identify degradation point and failure mode
Results Avg 890ms, P95 2.4s, 12.3% errors
Throughput Plateaus at ~950 req/sec
Finding: Connection pool exhaustion at ~150 users, PostgreSQL max_connections reached

Key Findings

  1. 1 Spin endpoint handles 780+ req/sec with sub-50ms average -- well within iGaming SLA requirements
  2. 2 Connection pool is the bottleneck -- reached max_connections at ~150 concurrent users
  3. 3 JWT validation adds negligible overhead (~2ms per request)
  4. 4 Database query optimization needed for /history endpoint under load (slowest P99)
  5. 5 Rate limiter (5 spins/sec per user) correctly throttles individual users without affecting overall throughput

Tool Apache JMeter 5.6.3 Protocol HTTPS with keep-alive connections Thread Groups setUp (auth) → load (mixed endpoints) → tearDown (report) Think Time Gaussian random timer (200-500ms) to simulate real user behavior Assertions Response code 200, JSON schema validation, response time threshold Data Unique user accounts per thread to avoid session conflicts Monitoring JMeter Summary Report + custom Groovy metrics collector
